摘 要： 随着计算机软件应用场景的不断增多， 软件开发面临诸多问题， 各类开发框架也逐渐出现运维、 部署、 耦合
度、 框架设计等方面的困难。针对以上问题， 提出通过微服务容器化实现开源社区整体架构的高效部署与自动化运
维， 以简化部署过程、 降低运维难度。通过介绍微服务与容器化技术的发展历程、 容器化关键技术与开源工具， 着重
论述微服务及容器编排工具在服务间通信与自动化运维等领域内的互补性， 分析并论证 Kubernetes 的优点与 Docker
的缺点， 以证明微服务容器化的优势。在应用部署方面， 以简单开源社区服务应用为例， 将传统单体架构应用改造成
微服务容器化服务以论述其优势和特点， 并论述了微服务容器化在移动边缘计算领域的发展前景与应用实例。最后
总结并讨论了微服务容器化的潜在问题及未来挑战。
关键词： 微服务； 开源社区； 容器化； 移动边缘计算

1 研究背景与现状
随着互联网的发展， 云计算与大数据、 人工智能等技
术推动了整个社会的信息化进程。但软件应用系统变得
更加复杂、 庞大且笨重， 原有框架已无法满足各类软件开
发过程中的部署及运维等方面的需求。因此， 众多企业和
机构通过反复探索与实践， 提出了多种开发框架模式以解
决上述问题。
在应用部署、 服务治理、 解耦及持续交付方面， 微服务
相对于传统框架有着诸多优势， 特别是持续交付
［1］： 由于
微服务架构对业务的拆解粒度更小， 使得集群运维变得更
加重要。于是学者们提出了容器化思想， 利用容器化技术
在部署及运维等方面的优势促进微服务容器化的发展
［2］。
1.1 微服务研究
在微服务问世之前， 软件工程师们提供了各种方式以
构建拥有广泛功能的系统 ——从早期的面向对象 ［3］的设
计模式、 面向服务的计算 （SOC） 、 面向服务的架构 （SOA） ，
到微服务概念的提出， 在此过程中服务粒度越分越细， 细
分到已无法保证系统架构在不断拆分中仍能将系统的复
杂度保持在一个较为合理的区间内
［4］。所以要通过一种
方式寻找微服务的边界， 以更好地划分服务， 提高开发效
［5］。除划分服务边界外， 微服务技术需要更具体的方案
及框架才能实现， 全球互联网公司都在尝试开发落地自己
的微服务技术。下面将介绍如何利用领域驱动设计划分
微服务组件与目前较为知名的微服务架构。
1.1.1 领域驱动设计
Evans 等［6］在 2004 年时提出一种用于业务功能领域建
模 的 框 架 与 思 想—— 领 域 驱 动 设 计 （Domain-Driven De‐
sign，DDD） 。该思想致力于实现的目标为： 交付令用户满
意、 可持续、 可不断演进的复杂软件。
DDD 设计分为两部分： 战略设计与战术设计。战略阶
段 强 调 从 宏 观 领 域 看 待 系 统 架 构［7］， 架 构 分 层 如 图1
所示［8］。
其次是战术设计阶段， 也即代码设计阶段。其使用聚
合、 实体、 值对象等对象概念表达和完善领域模型的细枝
［9］， 到这里就大致完成了DDD 设计。上文也提到微服
务在业务团队治理、 运维等方面有着先天优势， 但合理的
微服务粒度是实现微服务架构的前提
［5］。而 DDD 提出将
已划分好的限界上下文 （Bounded Context ，BC） 清晰业务边
界的概念引进微服务， 并以此实现了微服务框架。
1.1.2 微服务架构简介
相较于传统垂直架构， 微服务架构的具体优势及特点
如下［10］：
（1） 技术栈不限。相对于单体架构项目需要选择统一
的技术栈， 微服务架构可以将每个微服务应用作为一个独
立的应用进行维护。
（2） 扩展性强。可针对单点微服务应用的特点及需求
快速扩展， 以满足该节点的业务需求［11］。
（3） 可单独部署。相对于单体应用代码量大、 耦合度
高等特点， 单个应用进程独立且代码量小， 可实现独立、 高
［12］。
（4） 开发与维护方便。单个应用只需关注其特定的业
务功能， 通过定义好的接口来表述服务边界。
（5） 容错性强。某一服务发生故障， 运维人员可将故
障局限于该服务中进行修改， 并在其他节点中重启该应用
以继续提供服务。
微服务架构在日常开发维护、 容错性及弹性扩容等方
面 具 有 很 大 优 势 。 目 前 较 为 流 行 的 开 源 框 架 规 范 为
SpringCloud ， 其基于SpringBoot 对深层架构的再次封装， 为
开发人员提供了在分布式系统中快速构建微服务应用的
常用工具， 如配置管理、 服务发现、 断路器、 智能路由、 微代
理、 控制总线、 分布式会话、 集群管理等。另一种应用较多

1.1.3 微服务架构面临的挑战
一个大型互联网项目所需的微服务组件包括但不限
于： 服务网关、 注册中心、 配置中心、 服务集群、 消息队列、
分 布 式 存 储 、 分 布 式 搜 索 等 。 用 户 通 过 服 务 网 关 （Gate‐
Way） 接入整个业务体系。由于各微服务松耦合、 彼此访
问， 几乎所有的非业务功能如负载均衡、 服务认证、 访问限
制等均需服务网关来处理， 但缺点也很明显：
（1） 服务彼此依赖， 某一服务的异常都可能导致一部
分检测服务不可用［14］。
（2） 以人工为主的部署， 备份/可用性 /一致性差［15］， 无
法做到横向自动扩缩容。
因此， 只有探索软件平台能够实现服务部署、 自动化
运维、 弹性伸缩等特性的方案， 才能解决目前框架所遇到
的问题［16］。服务间尽可能减少依赖， 各自独立运行， 能够
实现高效访问， 因此推动了虚拟化技术， 特别是基于虚拟
化的容器技术的发展。随着该技术的蓬勃发展， 为软件开
发者提供了一种新思路。
1.2 容器技术
容器技术起源于虚拟化技术， 无论是传统意义上基于
KVM 的虚拟机还是容器， 均属于虚拟化的一种表现形式。
但相对于虚拟机， 容器更加轻便， 原因如下： 虚拟机需要在
物理机操作系统上运行自己的一系列系统进程 （ 需要独立
的操作系统， 系统进程与物理机完全隔离 ） ， 而其中产生的
除必要的业务组件外的性能损耗导致无法为专门的业务
应用配置一台虚拟机， 最终需要将多个应用组件分组塞入
不同的虚拟机中
［17］。虚拟机与容器对比如图 2 所示［18］。
容器的隔离机制会阻止一些容器内的进程监视另一
个容器内的进程 ［19］。同时， 这些容器服务不会干扰物理
机。 Felter 等［20］也通过 VM 与容器部署性能的对比， 证明
了在几乎所有情况下， 容器的性能与 VM 相同或更好。所
以通过容器技术将多个分散在不同物理机上的服务合而
为一的做法是可行的， 这也是众多数据中心采用该技术的
原因之一。此外， 容器技术还包含以下特性
［21-22］：
（1） 安全性。容器中可正常实现网络功能， 而由于容
器的隔离机制， 也可以限制一些危险操作。由于每个容器
之间是封闭的， 所以对单个容器的网络攻击不会影响其他
容器或系统进程。
（2） 隔离性。容器允许在一个物理机内运行多个应用
程序， 每个应用程序都会分配到单独的域以独享各自的资
源。与此同时， 也会给每个容器分配各自的 IP 地址， 在运
行多个应用时会使用统一的物理网络接口来保持应用 /服
务间的通信。
（3） 可移植性。容器将所需资源 （函数库、 环境等） 打
包成镜像， 利用容器的隔离机制可以将应用部署在任何地
方， 在具备便捷性的同时， 也兼具了灵活性。
（4） 速度。相对于虚拟机动辄几十秒甚至数分钟的启
动及停止时间， 容器的启动及停止时间只需几秒， 提高了
开发和运维工作效率， 提升了用户体验。
1.3 容器编排技术
随着容器化技术的发展， 开发人员可通过如 Docker、
LXC 等容器平台构建并运行镜像及容器， 但生产环境远比
开发及测试环境复杂。无论是传统架构还是微服务架构
的应用， 均涉及对一个或多个容器集群的管理， 而容器技
术对单个开发人员效率的提升并不能转化为对整个应用
集群管理效率的提升， 容器集群化需实现智能化、 自动化
［23］。而容器编排技术可以实现对容器进行自动化部
署、 管理、 扩展等操作，Zhong 等［24］详细讨论了将机器学习
用于容器编排方面的工作。使用容器编排可实现容器集
群 的 高 可 用 、 服 务 集 群 的 弹 性 伸 缩 、 负 载 均 衡 与 安 全
［25］。
容器编排工具也提供了用于大规模管理容器集群与
微服务架构的框架 ［25］， 代表产品有Docker Swarm 、Apache
Mesos 与 Kubernetes ， 本文将着重介绍Kubernetes 。
1.4 小结
如今， 随着云计算技术的发展和企业应用规模的扩
展， 企业应用逐渐从单体应用向微服务架构转换［26］。而传
统的微服务部署模式， 不仅复杂繁琐， 而且资源利用率低
下， 不利于扩展与更新。所以实现微服务部署的简化是一
个亟待解决的问题， 而容器技术提供了新的解决思路
［27］。
容器技术 +容器编排工具是国内外现阶段平台建设领域的
主流， 可通过使用容器化技术将每个服务实例运行在独立
的容器中
［16］。容器作为轻量级应用， 镜像构件和运行速度
极快， 甚至可在一个物理机或虚拟机上运行多个容器 ［28］。
与此同时， 使用容器编排工具对这些容器进行管理， 可充
容器编排工具对集群节点的自动化管理特性， 实现
业务服务器集群的自我修复、 弹性伸缩、 服务发现、 负载均
衡等功能。

态， 若状态不正常则重启容器。
（2）readiness probes 。其主要功能是测试当前应用能
否接收请求， 若不能则集群不会转发流量。
通过保证集群主节点的高可用和对容器集群内应用
的状态探测，K8s 有效保证了整个集群的高可用特性。
2.2 高效部署
如今微服务正逐步替代传统架构， 但服务的拆分使用
却给部署带来诸多问题： 在分布式架构系统中， 所依赖的
部件很多， 这些组件可能会引起一些冲突； 在部署时由于
各个服务器的环境差异也会遇到各种问题。特别是涉及
到互联网项目开发、 测试及生产环境的差异时， 会给微服
务部署带来极大困难。
可 利 用 容 器 化 技 术 解 决 这 些 问 题 ， 以Docker 为 例 ，
Docker 允许将开发中的应用、 依赖、 函数库、 配置一起打

隔离容器， 各个应用互不干扰， 且启动、 移除均可使用一行
命令完成。通过容器化技术以提高部署效率、 节约运维资
源， 为微服务集群的高效部署提供了有力支持。

3.1 Docker 架构简介
谈到容器技术时， 人们第一反应想到的大多是 Dock‐

Docker 使用的是客户端—服务器架构 （ C/S 架构） ， 组

以通过进程通信机制来管理 Docker。
（2）The Docker client 。用户可以通过命令行与 Docker
服务进行交互， 其使用Docker API ，Docker 客户端也可以与
其他守护进程进行通信。
（3）The Docker registries 。 用 于 存 储 Docker 镜 像 ，
Docker Hub 作为公共镜像库， 任何人都可以使用其中的镜
像。若有需求也可以建立自己的私人镜像库。
（4）The Docker objects 。当使用 Docker 时会涉及到创
建并使用镜像、 容器等对象， 这里介绍两个较为常用的对
象：①Images（镜像） ： 镜像将所需的各种环境依赖及系统
配置打包在一个只读的模版中， 需要时可以使用。除官方
镜像库提供的镜像外， 也可以使用 Dockerfile 构建自己的
镜像；②Containers（ 容器 ） ： 容器是镜像的可运行实例， 用户
可使用 Docker API 或 CLI 对容器进行各种操作， 且其对外
隔离性较好。

3.2.2 Kubernetes 微服务架构
Kubernetes 提供了当前容器技术体系内最好的微服务
架构解决方案， 具体特点如下［52］：
（1） 服务间通信机制。Kubernetes 架构中有一种用于
内部通信的设计， 名为Cluster IP ， 单一微服务应用通过对
外暴露接口的方式提供服务 （IP+端口） ， 只需知道该服务
的地址即可进行通信。 Kubernetes 通过给每个微服务分配
固定的 IP 地址 （Cluster IP） ， 同时引入DNS 机制， 将每个微
服务的名称作为域名与该 IP 捆绑， 客户端只需用服务名称
代替 IP 地址即可对该服务进行调用。
（2） 自动化。通常在一个大型微服务架构项目中有非
常多的微服务应用， 在某个应用中， 人工部署、 监控及扩容
一个微服务节点是很困难的， 可通过 Kubernetes 的 RC/De‐
ployment 对象结合容器高可用、 弹性扩容等特性实现微服
务的自动化。
（3） 持久化存储。 Kubernetes 通过独有的动态卷配置
（Dynamic Provision） 实现数据卷的自动创建、 部署、 回收等
策略， 而开发人员只需用PVC（Persistent Volumes Claim ） 描
述自己对持久化存储的需求， 并将创建好的PVC 挂载到需
要使用持久化存储的 Pod 上即可， 平台会根据PVC 描述的
条件自动分配合适的 PV 挂载到容器中。整个过程无需开
发人员关注， 一切由平台自动分配
［52］。
3.2.3 容器运行时
容器运行时 （Container Runtime） 指容器运行环境， 是
负责运行容器的软件［47］。Docker 就是一种容器运行时， 但
Docker 在问世之初并未考虑容器编排问题。直到 2014 年
Google 推出 Kubernetes 以解决 Docker 容器编排问题， 并在
随后的时间里逐渐成为业界主流的容器编排工具。
2020 年，Kubernetes 宣布在 1.20 版本废弃 Docker 作为
容器运行时 ［53］， 很多人认为Docker 要被淘汰了。其实不
必惊慌，Kubernetes 只是弃用 Docker 这个底层进行时， 转而
扶 持Kubernetes 自 己 创 建 的 容 器 运 行 接 口CRI（Container
Runtime Interface） 的 运 行 时 。 该 接 口 使 得kubelet 使 用 支
持 OCI 规 范 的 容 器 运 行 时 ， 而 无 需 重 新 编 译 ， 这 意 味 着
Docker 构建的镜像仍能正常运行在 Kubernetes 集群的运行
时中［54］。
Kubernetes 之所以放弃 Docker， 而不是将其集成到框
架中， 原因如下： 首先，Docker 不是一种单独的技术， 而是
一 个 完 整 的 技 术 栈 （containerd 只 是 Docker 中 的 一 个 组
件） ； 其 次 ，Docker 并 不 支 持cri，Kubernetes 为 兼 容Docker
不 得 不 引 入Dockershim 来 访 问Docker 内 部 的containerd ，
并造成许多不必要的系统性能损耗 ［54］。所以 Kubelet 跳过
Docker 直接调用 containerd ， 在保证架构稳定的同时， 又能
提升集群性能。因此， Kubernetes 1.20 弃用 Docker 作为容
器进行时。

3.3 Service Mesh
Service Mesh（ 服务网格 ） 是一个基础设施层， 用于处理
服务间通信。其将用户、 控制平面和数据平面彻底分离，
并为控制平面增加了诸多管控和诊断能力 ［56］。其解决了
传统微服务中如与特定语言绑定、 入门门槛较高、 业务逻

无侵入地添加多种功能， 避免了为满足第三方组件需求而
向应用添加额外的配置代码
［58］。简单来说， 作为服务间通
信的基础设施层， 业务团队只需关注与 Sidecar 的交互， 而
无需关注 Sidecar 与外部的交互， 比如与各种中间件或不同
语言的通信等［59］。

4 微服务实际应用
4.1 互联网应用： 电商平台
如今人们在现实生活中经常看到的 “视频带货” “直播
带货” ， 是随着前些年电商经济的发展而来的。随着用户
量激增， 各类需求增多， 传统单体垂直架构已无法满足电
商平台的现有需求
［64］， 此时需要使用微服务架构来满足海
量的高并发需求。
电商平台作为一种大型集群平台， 更注重于业务流的
稳定， 对整个业务集群的弹性伸缩、 可用性等方面要求很
高， 若集群某个业务节点在短时的高并发需求中停机， 整
个系统业务流程将无法运行。对于此类项目的微服务拆
分、 容器化部署及运维重点， 应从快速部署转到业务节点
之间的解耦合、 业务流程的合理性 （ 利用MQ 来异步处理业
务信息 ） 及维持主节点与业务节点集群的高可用上。
平台大体架构分为 3 层： 首先是微服务组件层， 使用
服务网关以实现网站的服务访问及部分负载均衡功能， 通
过注册中心使各个服务节点只需通过服务名称， 而无需具
体 IP 地址即可使用集群内服务； 其次是业务逻辑层， 通过
将各个具体业务拆解， 以实现单个节点的弹性伸缩和集群
的分布式管理， 同时使用消息队列异步访问应用， 以减轻
集群压力； 最后是数据访问层， 除搭建关系型数据库集群
外， 利用Redis 缓存大部分数据， 一定程度上解决了数据访
问的压力问题。
4.2 业务大数据应用： 科技资源共享
在每个企业及机构中， 一个业务系统往往由不同团队
进行开发， 随着时间推移， 在一次次更新后， 各个业务团队
所开发的业务模块和系统间的架构不尽相同， 而且涉及到
的数据等信息无法有效进行融合、 交换和分析， 微服务技
术的出现很好地解决了此问题
［65］。
在资源共享及注册方面， 各个开发团队只需将开发的
服务统一上传到 git 存储， 并将信息注册到服务中心， 即可
通过 Feign 来调用服务。而验证信息、 负载均衡方面则交
给 Zuul 网关来负责， 同时将已验证的信息放入 Redis 以减
少数据库负担。在系统部署发布方面， 利用 Jenkins 构建
Maven 项目， 只需指定好服务所在的git 地址，Maven 即可自
动将存储好的代码构建成项目， 并结合 Dockerfile 生成一
个微服务容器化镜像并运行。
4.3 基于微服务+容器技术的节能管理原型系统
随着云计算新技术的发展 （如 Serverless［66］等） ， 更多
机构选择将大部分本地数据及服务转移至云数据中心， 但
也使得数据中心的能耗增加， 导致其成为当地碳排放增长
最快的因素之一。 Xu 等［67］提出节能管理原型系统， 以评
估其提出的多层透视模型效果。该系统物理资源由 Open‐
Stack 管理， 而服务由Docker 管理， 该方案还通过Docker 将
单一应用打包进容器。而一台虚拟机可以有多个容器， 利
用容器编排工具 （该系统使用 Docker Swarm） 将这些容器
组合起来， 即可在单机中形成一个较为完整的微服务框
架。因此， 该系统可通过一系列方法评估各地数据中心的
能耗， 通过对数据中心的虚拟机进行整合及虚拟机数量缩
放实现节能减排的效果。
4.4 小结
以上方案致力于解决传统架构的问题， 特别是 SOA 架
构对于架构整体的严格限制。不同于 SOA 作为一个整体
应用部署， 且依靠ESB 来协调内部应用， 微服务更加强调
业务系统的组件化与服务化， 每个组件均能对外单独提供
服务， 且单个应用与容器结合紧密。因此， 在互联网应用
场景中， 微服务+容器技术显然更适应技术发展的趋势与
时代潮流。

5.2.1 外部访问 & 服务类型
在服务应用层， 部分模块端口需对外暴露以提供基础
服务， 而用户管理、Dashboard 模块端口对内使用， Kuber‐
netes 提供了一些 Service 类型以供选择［68］：
（1）Cluster IP 。为集群内部应用提供一个虚拟 IP， 设
定此类型后， 该应用只能在集群内部访问。
（2）NodePort。该服务节点暴露特定静态端口， Node‐
Port 将该节点的 Cluster IP 映射到暴露在外的端口上， 只需
输入本地机 IP 地址与对应服务的端口即可访问服务。
（3）LoadBalancer 。可以使用云服务商提供的负载均
衡服务器对外提供服务， 服务请求将会自动转发至 Node‐
Port 与 Cluster IP 上， 但其缺点在于每个对外服务都需要一
个 IP 地址和负载均衡服务器， 比较浪费。
（4）Ingress。Ingress 在严格意义上不属于一种 Service
服务， 其更像是一种智能路由或集群入口， 提供了负载均
衡、SSL 和虚拟托管等多种功能。 Ingress 也可基于子路径
或路由流量访问其他服务， 相对于 LoadBalancer 节省了很
多资源， 部署方式可参考官方文档［69］。Ingress 流量转发情
况如图 7 所示。
因该架构只用于测试环境， 所以选择 NodePort 对外提

6 微服务容器化在边缘计算系统中的应用
随着近几年的发展， 移动端设备数量不断增多、 性能
不断增强， 且随着5G 等现代通信技术的发展， 人们逐渐意
识到可以用云端来处理一些移动端无法处理的任务， 如数
值处理、 规则制定等， 于是便有了 “工业互联网” “车联网”
等新兴概念。本节将阐述微服务容器化如何在边缘计算
［72］落地应用， 并通过移动边缘计算、 车联网等应用场
景观察微服务容器化在边缘计算领域的发展前景。
6.1 边缘计算
边缘计算 （Edge Computing） 是一种分布式运算架构，
其将应用程序、 数据资料与服务的运算由网络中心节点移
往网络逻辑上的边缘节点进行处理 ［73-74］。相对于云计算，
边 缘 计 算 对 移 动 计 算 与 物 联 网 更 加 友 好 ， 其 具 有 以 下
优势［75］：
（1） 缓解网络带宽与数据中心压力。将大部分不需要
长期存储的数据放在网络边缘节点进行处理， 从而极大地
减轻了数据中心的压力。
（2） 增强服务的响应能力。利用边缘计算特性在用户
附近提供服务， 从而减少网络波动， 提高响应能力。
（3） 保护隐私数据。将私密数据限制在设备防火墙
中， 提升数据的安全性。
得益于这些优势， 边缘计算近些年来受到各个企业及
组织的关注， 得到了长足发展。但边缘计算模型与云计算
模型并不是互相取代的关系， 而是相辅相成的关系。边缘
计算需要数据中心强大的计算能力和海量存储的支持， 而
数据中心也需要边缘计算中边缘设备对海量数据及隐私
数据的处理
［76］。但边缘节点的性能往往不是很理想， 针对
边缘节点性能较差的特点， 也可利用微服务架构为边缘核
心节点提供一些服务。例如可充分利用开源边缘计算微
服 务 框 架EdgeX Foundry ， 其 广 泛 应 用 于 服 务 器 网 关 、 路
由、 边缘节点上。在此基础上，Lee 等
［77］在 2022 年提出可
在 Kubernetes 集群上运行 EdgeX Foundry ， 在保证服务间通
信的基础上， 进一步实现平台的智能化、 自动化运维［78］。
6.2 边缘计算在行业中的应用
移动边缘计算 （Moblie Edge Computing ， MEC） 通过字
面意思可知， 这项技术是用于移动通信系统下， 在离用户
很近的地方部署边缘计算节点， 并在节点上处理大量计算
传统的移动通信系统联网是将信号传送给附近基站，
基站将信号转换为数字信号后， 再一层层传到核心网， 最
后从核心网一层层传导到移动设备中。在网速还比较低、
对网络性能要求不高的时代， 这种层层转发的模式还是可
以接受的。但随着 5G 时代的到来， 对网络高带宽、 低时
延、 大容量方面的要求逐渐提高， 而解决问题的方式无非
两种： 增强网络性能或加强边缘节点计算能力
［79］。但网络
在一些极端情况下不是很可靠， 容易受到干扰， 在某些应
用场景如无人驾驶时遇到网络问题可能会出现严重事故。
所以人们选择了第二种方法， 主动推进未来移动网络逐渐
向 SDN/NFV、ICT 融合等技术框架方向演进。 MEC 技术在
此 背 景 下 应 运 而 生 ， 实 现 了 网 络 与 业 务 的 深 度 融 合［80］。

本文论述了微服务和容器化相关技术方案及其特点
与逻辑关系， 将单体架构应用进行微服务容器化改造， 并
系统介绍了现有开源工具及其发展路线和未来趋势。详
细介绍了开发运维过程中的四大关键技术， 并结合开源社
区微服务项目讨论开发运维细节。通过研究微服务容器
化在典型应用中的使用场景， 以便更好地将微服务容器化
延伸到现实应用中。随后自行搭建了轻量级开源社区应
用， 通过实际拆解原始架构、 设计组件及部署容器化应用，
深刻体会到微服务容器化不仅在理论层面上具有研究价
值， 而且在应用层面上具有很强的应用价值。但其中仍有
一些潜在问题与挑战需作进一步讨论：
（1） 容器化集群虽然解决了微服务架构的部署问题，
但容器集群管理工具的部署本身也带来了很多问题。例
如，Kubernetes 部署复杂且对系统性能有一定要求， 而官方
提供的 minikube 最低也需要 2GB 内存以运行该工具。即
便利用 K3s 代替 Kubernetes 部署也存在诸如大型系统稳定
性等问题， 也需作进一步研究。
（2） 在服务迁移方面， 针对不同的应用场景、 系统架构
特点， 对部署的方式、 规模、 数据迁移及后期维护等方面提
出了很大挑战。例如， 部署容器时是拉取公共镜像库还是
自行构建应用镜像， 持久化存储以何种方式迁移并应用，
以及平台所需的物理机数量与性能等均需加以考虑。
（3） 相对于传统单体架构易被入侵， 由于服务网关的
存在， 微服务架构在数据安全方面有着巨大优势。而容器
编排工具在此基础上更进一步， 如 Kubernetes 将每个微服
务的容器封装到 Pod 中， 外界只能通过Service 对外暴露的
接口访问 Pod， 不能直接访问Pod 内的容器， 而Pod 内的容
器可互相访问， 可通过将相近服务部署在同一 Pod 的方式
保证单一应用及容器集群的安全性。
（4） 在平台标准制定与行业合作方面有待加强， 虽然
企 业 在 搭 建 平 台 时 参 考 了 如SpringCloud 、Istio 等 开 源 框
架， 但接口及部署方式均由各个企业自行定义， 影响了框
架迁移及数据的共享效率与可行性。在对各个企业的需
求进行定制化设计的同时， 如何规定统一的 API 接口等都
是需要后期持续关注并研究的问题。
